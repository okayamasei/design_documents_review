# テーブル定義書1
## 全体
- テーブルの名前が単数形になっているところがある
	>テーブル名は複数形
- テーブルの名前の頭文字が小文字じゃないところがある
	>テーブル名は全てアルファベットの小文字
- テーブルの名前がスネークケースになっていないところがある
	>テーブル名はスネークケースで書く
- created_atが全てのテーブルでユーザ登録日時になってしまっている
- updated_atが全てのテーブルでユーザ更新日時になってしまっている
	>全テーブルでcreated_atがユーザー登録日時に、updated_atがユーザー更新日時となっているが、Usersテーブル以外では命名を変える必要がある
- id カラムがない
	>全てのテーブルに対して"id"カラムが自動で作成されるので、抜けている部分は追加する必要がある
- カラム名の最初にテーブル名がついてしまっているものがある
	>接頭辞にテーブル名が付いているカラムが多数あるので見直しましょう

## Admin
- PKがない
	>PKはどのカラムとなりますか

## Users
- 管理者側から見た適切な名前じゃない。
	>end_usersなどが適切な名前です。適切な命名をしましょう
- 配送先住所がある
	>住所と別の配送先は、一つしか登録できないのでしょうか
	>複数配送先を登録する場合、addressesテーブルでアドレスを管理する必要があります
- member_statusのデータ型がおかしい
	>trueかfalseで管理する場合はboolean型が便利です
- member_statusのカラム名が適切じゃない
	>member_statusだと、何のステータスを表しているか分かりづらいです
- 電話番号がinteger型になっている
- 郵便番号がinteger型になっている
	>integer型だと先頭の0が保存されないので適切ではありません。また、計算に扱うカラムではないことも適切でない理由の１つです。
	>適切なデータ型にしましょう
- 名前（性）、名前（名）、名前(性カナ)、名前(名カナ)のカラム名は略語を使わない
	>省略して書くと、他の人が見たときにわかりにくいので、わかりやすい命名にしましょう
	>また、性ではなく姓が正しい

## Products
- FKが書かれていない
	>他のテーブルのFKを持ってきている場合、FKの欄に丸をつけましょう
- cd_imageとcd_titleは頭のcd_が付いている
	>なんのタイトルや画像かはテーブル名から明らかなのでいらないです
- product_という接頭辞がついてる
	>主キーに接頭辞は不要です
- ジャケット画像は、refileを使う場合、データ型はstringで、NOT NULLは○にする
	>また、カラム名は○○_idとする
	>*商品画像はあくまでも画像idなので、kの命名が適切
- 商品名、発売日のカラムがない
	>商品名と発売日はどこで管理しますか

## Discs
- products_idはsいらない
	>FKにはdiscに紐づいている一件のproductのidが入るので、product_idに変更しましょう

## Songs
- 曲名を入れるであろうカラムがinteger型になっている
	>適切なデータ型を見直しましょう
- 曲名を入れるカラムの名前が不明確
	>nameなど、わかりやすい命名をしましょう
- PKがない
	>主キーにPKを書きましょう

## Labels
- レーベル名を保存するカラムの名前が不明確
	>nameなど、わかりやすい命名をしましょう

## Artists
- アーティスト名のカラムのデータ型がintegerになっている
	>アーティスト名は文字も入るはずです。適切なデータ型を見直しましょう
- アーティスト名を保存するカラムの名前が不明確
	>nameなど、わかりやすい命名をしましょう

## Genre
- ジャンル名を保存するカラムの名前が不明確
	>nameなど、わかりやすい命名をしましょう
- ジャンル名のカラムのデータ型がintegerになっている
	>ジャンル名には文字も入るはずです。適切なデータ型を見直しましょう

## Cart item
- products_idはsいらない
	>FKとして持たせる場合、単数形にしましょう
- buy num の命名が不適
	>複数の単語を使う場合はスネークケースにしましょう
	>省略して命名しないようにしましょう。他の人が読んでもわかるように命名するのが適切です

## Buy detail
- Buyは購入という意味なので、名前が適切ではない
	>受注の意味になるように命名しましょう
- buy numの命名が不敵
	>複数の単語を使う場合はスネークケースにしましょう
	>省略して命名しないようにしましょう。他の人が読んでもわかるように命名するのが適切です
- 外部キーにNOT NULLとデータ型がない
	>適切なデータ型を設定しましょう
	>NULLを許容する場合、NOT NULLに丸をつけましょう

## Buy
- Buyは購入という意味なので、名前が適切ではない
	>受注の意味になるように命名しましょう
- stockの命名が不敵
	>カラム名から用途が想像できません。わかりやすい命名に直しましょう
	>支払い方法は、payment_methodと表現をする
- 支払い方法は３パターンあるのでenumで管理するが、データ型がstringになっている
	>string型で管理する場合、方法の文字を変えようとした際に大量のデータをアップデートしなければいけなくなります。また、全角文字を使うとデータサイズが大きくなり、不便です
	>enumで調べて適切に直しましょう
- 配送ステータスを管理するカラムがない
	配送ステータスはどのように管理しますか
